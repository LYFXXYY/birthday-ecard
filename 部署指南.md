# 生日贺卡系统部署指南

## 一、部署技术入门

> 本章面向零基础读者。如果你已有部署经验，可以跳过。

### 1.1 什么是"部署"

开发时，代码运行在你自己的电脑上，只有你能访问。**部署**就是把代码放到一台联网的服务器上，让其他用户能通过浏览器或手机访问你的程序。

打个比方：开发像在自家厨房做菜，部署就是把菜端上餐厅让客人品尝。你需要一个"厨房"（服务器）、一套"上菜流程"（部署工具）、一个"门面"（域名或 IP 地址）。

### 1.2 Node.js 项目部署核心概念

**进程管理** — 开发时用 `node src/app.js` 启动，关掉终端就停了。生产环境需要 **PM2**（Process Manager 2）这个"管家"，让服务在后台持续运行、崩溃自动重启、开机自动启动。

**反向代理** — Node.js 默认监听 3000 等非标准端口，但用户访问的是 80（HTTP）或 443（HTTPS）。**Nginx**（读作 "engine-x"）作为"前台接待"，接收用户请求后转发给 Node.js：

```
用户浏览器 ──> Nginx (80/443端口) ──> Node.js (3000端口)
```

**环境变量** — 数据库密码、API 密钥等敏感信息放在 `.env` 文件中，不写入代码，也不提交到 Git。

**HTTPS** — 生产环境应使用加密传输，通常通过 Nginx 配置 SSL 证书实现（可用 Let's Encrypt 免费证书）。

### 1.3 Spring Boot 部署对比（拓展认知）

了解 Java Spring Boot 的部署方式有助于建立全局认知：

| 对比项 | Node.js (本项目) | Spring Boot |
|--------|------------------|-------------|
| 打包方式 | 直接运行源码 | 编译打包为 JAR 文件（`java -jar app.jar`） |
| Web 服务器 | Express 需搭配 Nginx | 内嵌 Tomcat，自带 Web 服务器 |
| 进程管理 | PM2 | systemd（Linux 内置服务管理器） |
| 依赖安装 | 服务器上 `npm install` | JAR 自包含所有依赖 |

### 1.4 常用部署工具简介

| 工具 | 用途 | 一句话解释 |
|------|------|-----------|
| **PM2** | 进程管理 | 让 Node.js 后台持续运行，崩溃自动重启 |
| **Nginx** | 反向代理 + 静态文件 | 接收用户请求并转发，高效提供前端页面 |
| **Docker** | 容器化部署 | 把应用和依赖打包成"集装箱"，任意机器一键运行（本项目暂不使用，了解概念即可） |

## 四、完整部署 — Linux（生产环境推荐）

### 4.1 服务器准备

```bash
# 安装 Node.js（以 Ubuntu 为例）
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs nginx mysql-server

# 安装 PM2 进程管理器
sudo npm install -g pm2
```

### 4.2 数据库配置

```sql
CREATE DATABASE birthday CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'birthday_app'@'localhost' IDENTIFIED BY '强密码_至少16位';
GRANT ALL PRIVILEGES ON birthday.* TO 'birthday_app'@'localhost';
FLUSH PRIVILEGES;
```

### 4.3 后端部署

```bash
git clone <repository_url> /opt/birthday-card-system
cd /opt/birthday-card-system/server
npm install --production       # --production 跳过开发依赖
cp .env.example .env && nano .env
```

**.env 生产配置要点：**

```env
# 数据库（使用专用用户，不要用 root）
DB_HOST=localhost
DB_PORT=3306
DB_NAME=birthday
DB_USER=birthday_app
DB_PASSWORD=强密码_至少16位

# JWT（必须改为随机长字符串！生成方法: openssl rand -base64 48）
JWT_SECRET=至少64位的随机字符串
JWT_EXPIRES_IN=24h

# 服务器
PORT=3000
NODE_ENV=production
LOG_LEVEL=info                 # 生产环境不要设 debug

# 短信（生产环境改为 carrier）
SMS_PROVIDER=carrier
SMS_API_URL=你的运营商接口地址
SMS_API_KEY=你的API密钥
SMS_API_SECRET=你的API密钥

# CSP 5G视频短信
CSP_APP_ID=你的应用ID
CSP_PASSWORD=你的应用密码
CSP_VIDEO_TEMPLATE_ID=审核通过的模板ID
```

```bash
# 使用 PM2 启动并设置开机自启
pm2 start src/app.js --name birthday-backend
pm2 startup          # PM2 会输出一条命令，复制执行
pm2 save
```

### 4.4 前端部署

```bash
# 在本地电脑构建（不需要在服务器上装前端依赖）
cd admin-web && npm install && npm run build
# 上传到服务器
scp -r dist/* user@server:/var/www/birthday-admin/
```

### 4.5 Nginx 配置

```bash
sudo nano /etc/nginx/sites-available/birthday-admin
```

```nginx
server {
    listen 80;
    server_name admin.yourcompany.com;  # 改为你的域名或服务器IP

    root /var/www/birthday-admin;
    index index.html;

    # 前端 history 模式（刷新不 404）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 代理
    location /api/ {
        proxy_pass http://localhost:3000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 贺卡和视频资源代理
    location /cards/  { proxy_pass http://localhost:3000/cards/; }
    location /videos/ { proxy_pass http://localhost:3000/videos/; }
}
```

```bash
sudo ln -s /etc/nginx/sites-available/birthday-admin /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

### 4.6 监控项目（可选独立部署）

```bash
cd /opt/birthday-card-system/server/monitor
npm install --production
pm2 start src/app.js --name birthday-monitor && pm2 save
```

### 5.4 使用 NSSM 注册为 Windows 服务

NSSM（Non-Sucking Service Manager）把 Node.js 注册为 Windows 服务，实现开机自启和崩溃重启。

```powershell
# 下载 NSSM：https://nssm.io/builds，解压到 C:\nssm\

# 注册服务（管理员 PowerShell）
C:\nssm\nssm.exe install birthday-backend
# GUI 界面中填写：
#   Path:        C:\Program Files\nodejs\node.exe
#   Arguments:   C:\birthday-card-system\server\src\app.js
#   Startup dir: C:\birthday-card-system\server

C:\nssm\nssm.exe start birthday-backend

# 常用管理
C:\nssm\nssm.exe status birthday-backend     # 查看状态
C:\nssm\nssm.exe restart birthday-backend    # 重启
C:\nssm\nssm.exe stop birthday-backend       # 停止
```

> **备选**：也可以用 PM2（`npm install -g pm2`），但 Windows 上开机自启不如 NSSM 稳定。

### 5.5 前端部署

```powershell
# 本地构建后复制到 IIS Web 目录
cd admin-web && npm install && npm run build
Copy-Item -Recurse dist\* C:\inetpub\wwwroot\birthday-admin\
```

### 5.6 使用 IIS 做反向代理

**启用 IIS 并安装 ARR：**

```powershell
Enable-WindowsOptionalFeature -Online -FeatureName IIS-WebServerRole
# 安装 ARR 3.0 + URL Rewrite：https://www.iis.net/downloads/microsoft/application-request-routing
# 安装后在 IIS 管理器中启用 "Proxy" 功能
```

**在 `C:\inetpub\wwwroot\birthday-admin\` 下创建 `web.config`：**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <rule name="API Proxy" stopProcessing="true">
          <match url="^api/(.*)" />
          <action type="Rewrite" url="http://localhost:3000/api/{R:1}" />
        </rule>
        <rule name="Cards Proxy" stopProcessing="true">
          <match url="^cards/(.*)" />
          <action type="Rewrite" url="http://localhost:3000/cards/{R:1}" />
        </rule>
        <rule name="Videos Proxy" stopProcessing="true">
          <match url="^videos/(.*)" />
          <action type="Rewrite" url="http://localhost:3000/videos/{R:1}" />
        </rule>
        <rule name="SPA Fallback" stopProcessing="true">
          <match url=".*" />
          <conditions>
            <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
            <add input="{REQUEST_FILENAME}" matchType="IsDirectory" negate="true" />
          </conditions>
          <action type="Rewrite" url="/index.html" />
        </rule>
      </rules>
    </rewrite>
  </system.webServer>
</configuration>
```

> **备选**：也可下载 Nginx for Windows（https://nginx.org/en/download.html），配置与 Linux 版相同（见 4.5 节），路径改为 Windows 格式即可。

### 5.7 防火墙配置

```powershell
# 开放 80/443 端口（通常已存在，确认即可）
New-NetFirewallRule -DisplayName "HTTP" -Direction Inbound -Port 80 -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "HTTPS" -Direction Inbound -Port 443 -Protocol TCP -Action Allow
# 不要开放 3000 端口给外网，Nginx/IIS 负责对外转发
```

### 5.8 数据库备份（Windows 计划任务）

```powershell
# 创建备份脚本 C:\birthday-card-system\backup.ps1：
$backupDir = "C:\birthday-card-system\backups"
$date = Get-Date -Format "yyyyMMdd"
if (!(Test-Path $backupDir)) { New-Item -ItemType Directory -Path $backupDir }
mysqldump -u birthday_app -p'你的密码' birthday | Out-File "$backupDir\birthday_$date.sql" -Encoding UTF8
Get-ChildItem "$backupDir\*.sql" | Where-Object { $_.CreationTime -lt (Get-Date).AddDays(-30) } | Remove-Item

# 创建计划任务（每天凌晨 2 点）
$action = New-ScheduledTaskAction -Execute "PowerShell.exe" -Argument "-ExecutionPolicy Bypass -File C:\birthday-card-system\backup.ps1"
$trigger = New-ScheduledTaskTrigger -Daily -At 2:00AM
Register-ScheduledTask -TaskName "BirthdayDBBackup" -Action $action -Trigger $trigger
```

### 5.9 Windows 注意事项

- **路径分隔符**：代码中使用 `path.join()` 而非手动拼接，Node.js 自动处理分隔符
- **PowerShell vs CMD**：推荐 PowerShell；CMD 中命令语法不同（如 `copy` 而非 `Copy-Item`）
- **后台进程**：不要直接在 CMD 窗口运行 `node src/app.js`，关窗口就停了，务必用 NSSM 或 PM2
- **文件权限**：确保 Node.js 进程对 `generated-cards/`、`generated-videos/`、`logs/` 有写入权限
- **换行符**：建议在项目根目录创建 `.gitattributes`（内容 `* text=auto`），避免 Git 把 LF 转为 CRLF

## 六、Docker 容器化部署介绍

> **本项目当前不使用 Docker，此章节仅供了解概念。** 如果未来项目规模扩大或需要跨环境部署，可以考虑引入 Docker。

### 6.1 什么是 Docker

**一句话概括**：Docker 是一种把应用程序和它所需的全部依赖（运行环境、库、配置文件等）打包在一起的工具，打包后的东西可以在任何安装了 Docker 的电脑上运行。

**集装箱比喻**：国际贸易中，货物要经过轮船、火车、卡车多次转运。过去每种运输方式都要重新装卸，效率低且容易出错。后来发明了**集装箱**——不管里面装什么，外面的尺寸统一，任何运输工具都能直接搬运。Docker 就是软件世界的"集装箱"：

| 现实世界 | Docker 世界 |
|---------|------------|
| 集装箱 | **容器（Container）** — 运行中的应用实例 |
| 集装箱设计图纸 | **镜像（Image）** — 构建容器的模板/快照 |
| 工厂的生产流水线 | **Dockerfile** — 编写镜像构建步骤的配方文件 |
| 物流调度中心 | **Docker Compose** — 同时管理多个容器的编排工具 |

**容器 vs 虚拟机**：虚拟机（如 VMware）模拟了一整台电脑，包含完整的操作系统，启动慢、占资源多。容器共享宿主机的操作系统内核，只打包应用层面的差异，启动只需几秒，资源占用小得多。打个比方：虚拟机像在每层楼都建一个独立的锅炉房供暖，容器像集中供暖——共享一个锅炉（内核），每个房间（容器）只装暖气片。

### 6.2 Docker 的优势与适用场景

**优势：**

- **环境一致性**：彻底解决"在我电脑上能跑啊"的问题。开发、测试、生产环境使用同一个镜像，行为完全一致
- **快速部署与回滚**：`docker pull` 拉取镜像 → `docker run` 启动，几秒钟完成部署。出问题？切回上一个镜像即可回滚
- **隔离性好**：每个容器互不干扰，一个崩溃不影响其他服务
- **适合微服务架构**：每个服务独立打包、独立部署、独立扩缩容

**本项目暂不使用 Docker 的原因：**

- 本项目是**单体应用**（前后端 + 数据库），直接部署更简单直观
- 主要部署环境是 **Windows Server**，Docker Desktop 在 Windows 上有额外限制（见 6.3 节）
- 团队对 Docker 尚不熟悉，引入会增加学习和维护成本
- 当前部署方式（PM2/NSSM + Nginx/IIS）已能满足需求

### 6.3 Windows 上安装 Docker

如果想在 Windows 上体验 Docker，需要满足以下条件：

**系统要求：**

- Windows 10/11 **Pro / Enterprise / Education** 版本（Home 版不支持）
- 需要启用 **WSL2**（Windows Subsystem for Linux 2）
- BIOS 中开启虚拟化（VT-x / AMD-V）
- 至少 4GB 内存（推荐 8GB+）

**安装步骤：**

```powershell
# 1. 启用 WSL2（管理员 PowerShell）
wsl --install
# 安装完成后重启电脑

# 2. 确认 WSL2 版本
wsl --version
# 应显示 WSL 版本 2

# 3. 下载安装 Docker Desktop
# 下载地址：https://www.docker.com/products/docker-desktop/
# 安装时选择 "Use WSL 2 instead of Hyper-V"
# 安装完成后重启电脑

# 4. 启动 Docker Desktop 并验证
docker --version
docker run hello-world    # 运行测试容器，成功会打印欢迎信息
```

**Windows 特有注意事项：**

- **Hyper-V 冲突**：Docker Desktop 早期版本使用 Hyper-V，与 WSL2 后端冲突。新版已支持 WSL2 后端（推荐），安装时注意选择
- **资源占用**：Docker Desktop 常驻内存约 2-4GB，低配机器可能卡顿。可在设置中限制 CPU 和内存上限
- **文件系统性能**：跨文件系统（在 Windows 容器中访问 Windows 路径）较慢，建议将项目文件放在 WSL2 的 Linux 文件系统中
- **Windows Home 版**：不支持 Docker Desktop。替代方案是先安装 WSL2，再在 WSL2 的 Linux 环境中安装 Docker Engine

### 6.4 本项目如果用 Docker 部署（概念示意）

> 以下为概念性描述，帮助理解 Docker 的部署思路。本项目当前不提供 Dockerfile。

**三个容器的架构：**

```
┌──────────────────────────────────────────────────┐
│                 Docker 主机                        │
│                                                  │
│  ┌──────────┐   ┌──────────────┐   ┌──────────┐ │
│  │  MySQL    │   │  Node.js     │   │  Nginx   │ │
│  │  容器     │<──│  后端容器     │<──│  容器    │ │
│  │  :3306   │   │  :3000       │   │  :80     │ │
│  └──────────┘   └──────────────┘   └──────────┘ │
│       ▲               ▲               │         │
│       │          ┌────┴────┐          │         │
│       │          │ Volumes │          │         │
│       │          │ (持久化) │          │         │
│       │          └─────────┘          │         │
│  ┌────┴────────────────────────────┐   │         │
│  │        宿主机端口映射            │<──┘         │
│  │  3306→MySQL  3000→API  80→Web  │             │
│  └─────────────────────────────────┘             │
└──────────────────────────────────────────────────┘
```

**docker-compose.yml 大致结构（仅示意）：**

```yaml
# 这个文件定义了三个容器如何协同工作
services:
  db:                        # MySQL 数据库容器
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: xxx
      MYSQL_DATABASE: birthday
    volumes:
      - db-data:/var/lib/mysql    # 数据持久化到宿主机

  backend:                   # Node.js 后端容器
    build: ./server          # 用 server/ 目录下的 Dockerfile 构建
    depends_on:
      - db                   # 依赖数据库先启动
    environment:
      DB_HOST: db            # 容器间通过服务名通信
      DB_PORT: 3306
    volumes:
      - card-data:/app/generated-cards   # 贺卡文件持久化
      - video-data:/app/generated-videos # 视频文件持久化

  frontend:                  # Nginx + 前端静态文件容器
    image: nginx:alpine
    ports:
      - "80:80"              # 宿主机 80 端口映射到容器
    depends_on:
      - backend

volumes:                     # 声明持久化数据卷
  db-data:
  card-data:
  video-data:
```

**数据卷（Volumes）** 是 Docker 持久化数据的机制。容器本身是"一次性"的——删除容器，里面的数据也会丢失。数据卷将容器内的目录映射到宿主机的存储中，确保数据库文件和生成的贺卡视频不会因容器重启而丢失。

### 6.5 Docker 常用命令速查

**镜像操作：**

```bash
docker build -t myapp:1.0 .       # 用当前目录的 Dockerfile 构建镜像，命名为 myapp:1.0
docker pull mysql:8.0              # 从 Docker Hub 拉取 MySQL 8.0 镜像
docker images                      # 列出本地所有镜像
docker rmi myapp:1.0               # 删除镜像
```

**容器操作：**

```bash
docker run -d -p 3000:3000 myapp   # 后台运行容器，映射端口
docker ps                            # 查看正在运行的容器
docker ps -a                         # 查看所有容器（含已停止的）
docker stop <容器ID>                 # 停止容器
docker start <容器ID>                # 启动已停止的容器
docker logs <容器ID>                 # 查看容器日志
docker logs -f <容器ID>              # 实时跟踪容器日志（类似 tail -f）
docker exec -it <容器ID> bash        # 进入容器内部执行命令（调试用）
docker rm <容器ID>                   # 删除容器
```

docker-compose logs -f backend       # 实时跟踪指定服务的日志
docker-compose restart backend       # 重启指定服务
docker-compose ps                    # 查看各容器状态
docker-compose pull                  # 拉取最新镜像

## 九、故障排查

### 通用问题

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| 后端启动失败 | 端口 3000 被占用 | Linux: `lsof -i:3000` 后 `kill` 占用进程；Windows: `netstat -ano \| findstr :3000` 查 PID，`taskkill /PID <pid> /F` |
| 数据库连接失败 | 密码错误或权限不足 | 检查 `.env` 数据库配置，确认用户有对应数据库权限 |
| 前端白屏 | 反向代理未生效 | 检查 Nginx/IIS 代理配置，浏览器 F12 看 Network 面板 |
| 监控进程退出 | 后端崩溃 | 检查后端日志，修复后重启监控 |
| 短信发送失败 | CSP 配置或网络问题 | 检查 `.env` 中 CSP 配置，确认服务器能访问外网 |
| npm install 失败 | 网络或版本问题 | 尝试淘宝镜像：`npm config set registry https://registry.npmmirror.com` |

### Windows 特有问题

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| 80 端口被占用 | IIS 默认网站占用 | IIS 管理器中停止 "Default Web Site" 或修改绑定端口 |
| NSSM 服务启动后立即停止 | 路径错误或配置有误 | `nssm edit birthday-backend` 检查配置；查看 I/O 选项卡中的错误日志 |
| 权限拒绝（EACCES） | 无写入权限 | 右键文件夹 → 属性 → 安全 → 添加 SYSTEM 完全控制权限 |
| 路径找不到文件 | 分隔符或空格问题 | 使用 `/` 或 `\\`；含空格路径用引号包裹 |
| PowerShell 无法运行脚本 | 执行策略限制 | 管理员运行 `Set-ExecutionPolicy RemoteSigned -Scope CurrentUser` |
| MySQL 服务未启动 | 未设置自动启动 | `services.msc` 中找到 MySQL，设为"自动"并启动 |
| `node` 不是内部命令 | 未加入 PATH | 重新安装 Node.js 勾选 "Add to PATH"，或手动添加环境变量 |
