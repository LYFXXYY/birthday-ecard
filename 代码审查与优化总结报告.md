## 一、已识别并修复的问题

### 1.1 P0 - 阻塞性/严重安全问题（全部已修复）

#### 问题1: SendRecord模型缺少 `admin_id` 字段
- **严重程度**: 高 (数据丢失)
- **描述**: `models/index.js` 定义了 `SendRecord.belongsTo(Admin, { foreignKey: 'admin_id' })`，且 `employees.js` 手动生成贺卡时写入 `admin_id`，但 `SendRecord` 模型定义中没有该字段，Sequelize 会静默丢弃数据
- **修复**: 在 `SendRecord.js` 模型中添加了 `admin_id` 字段定义

#### 问题2: records路由缺少认证中间件
- **严重程度**: 高 (安全漏洞)
- **描述**: 其余路由（employees, templates）都使用了 `router.use(authMiddleware)`，唯独 `records.js` 遗漏，发送记录和统计数据无需认证即可访问
- **修复**: 在 `records.js` 路由中添加 `router.use(authMiddleware)`

#### 问题3: response.js 的 `error()` 函数参数设计混乱
- **严重程度**: 高 (数据错误)
- **描述**: 旧签名 `error(res, message, code, statusCode)` 四个参数容易混淆，`records.js` 中调用 `error(res, '员工不存在', 1, 404)` 导致业务码为1而非404，`employees.js` 中 `error(res, '部分验证失败', 400, {...})` 将对象传入了statusCode位置
- **修复**: 简化为 `error(res, message, statusCode, extraData)` 三个核心参数

#### 问题4: 调度器创建失败记录时 `template_id: null`
- **严重程度**: 高 (数据库约束违反)
- **描述**: `scheduler.js` 异常处理中创建失败记录时 `template_id: null`，但模型定义 `allowNull: false`，导致数据库约束错误
- **修复**: 重构异常处理逻辑，失败时仅记录日志，不创建不完整的数据库记录

#### 问题5: 模板CRUD存在Mass Assignment漏洞
- **严重程度**: 高 (安全漏洞)
- **描述**: `Template.create(req.body)` 和 `Template.update(req.body)` 直接将请求体传递给数据库操作，攻击者可通过请求篡改任意字段
- **修复**: 添加 `sanitizeInput()` 白名单过滤函数

#### 问题6: 贺卡路由存在路径遍历风险
- **严重程度**: 高 (安全漏洞)
- **描述**: card路由仅对cardId做了长度检查(>=10)，未防范 `../` 等路径遍历字符
- **修复**: 使用UUID正则验证格式 + `path.basename()` + `path.resolve()` 防止目录穿越

#### 问题7: Vite 代理配置三处错误（阻塞性）
- **严重程度**: 高 (开发环境无法启动)
- **位置**: `admin-web/vite.config.ts`
- **修复**: 代理目标改为 `http://localhost:3000`，前端端口改为 `5173`，移除 `rewrite` 规则

#### 问题8: Express 路由顺序导致 `today-birthday` 不可达
- **严重程度**: 高 (功能不可用)
- **位置**: `server/src/routes/employees.js`
- **修复**: 将 `GET /today-birthday` 路由移到 `GET /:id` 之前

#### 问题9: FormData Content-Type 边界丢失
- **严重程度**: 高 (功能不可用)
- **位置**: `admin-web/src/api/employees.ts`
- **修复**: 移除手动设置的 Content-Type 头，让浏览器自动处理

#### 问题10: Dashboard 组件硬编码模拟数据
- **严重程度**: 高 (数据不准确)
- **位置**: `admin-web/src/views/Dashboard.vue`
- **修复**: `isDevMode` 改为环境变量控制，员工总数改为调用真实 API，字段名统一为 snake_case

#### 问题11: 模板预览功能被 JSON 拦截器破坏
- **严重程度**: 高 (功能不可用)
- **位置**: `admin-web/src/api/request.ts` + `admin-web/src/api/templates.ts`
- **修复**: 响应拦截器添加 Content-Type 检测，`text/html` 响应直接返回原始数据

#### 问题12: 员工字段白名单遗漏 snake_case 形式
- **严重程度**: 高 (数据丢失)
- **位置**: `server/src/routes/employees.js`
- **修复**: 白名单同时包含两种形式，并在 `sanitizeEmployeeInput` 中统一映射

### 1.4 短信服务与模板系统改造（v1.4.0 新增）

#### 问题28: 短信发送服务为孤立桩函数，未集成到业务流程
- **描述**: `smsService.js` 仅模拟发送且从未被任何模块调用，`scheduler.js` 生成贺卡后直接标记为成功，跳过了短信发送环节
- **修复**: 
  - 重写 `smsService.js`，实现 Provider 抽象（mock/carrier 双模式）、指数退避重试机制、结构化错误处理
  - 集成到 `scheduler.js` 定时任务：生成贺卡 -> 创建 pending 记录 -> 发送短信 -> 更新记录状态
  - 集成到 `employees.js` 手动发送端点和 `records.js` 测试发送端点
  - 安装 axios 作为 HTTP 客户端依赖
  - 在 `config/index.js` 和 `.env.example` 中添加 SMS 配置项

#### 问题29: SendRecord 模型缺少短信发送追踪字段
- **描述**: 无法追踪短信是通过哪个 Provider 发送的、重试了几次、运营商返回了什么消息ID
- **修复**: 添加 `message_id`（运营商消息ID）、`sms_provider`（mock/carrier）、`retry_count`（重试次数）三个字段

#### 问题30: 贺卡模板缺少 `{{position}}` 和 `{{birthday}}` 占位符支持
- **描述**: 前端 `card_template/` 中的3个生产级模板使用了 `{{position}}` 和 `{{birthday}}` 占位符，但 `cardGenerator.js` 只处理了4个占位符（name/department/blessing/year），导致这两个字段渲染为原始占位符文本
- **修复**: 在 `cardGenerator.js` 中添加 `{{position}}` 和 `{{birthday}}` 替换，生日格式化为 "M月D日"

#### 问题31: 贺卡模板系统架构升级（播种器 + 占位符扩展）
- **描述**: 原有 `initDefaultTemplate.js` 只导入单一默认模板，且 `cardGenerator.js` 仅支持4个占位符（name/department/blessing/year），缺少 position 和 birthday
- **修复**: 
  - 从 `admin-web/card_template/` 复制3个正确模板到 `server/src/data/`（template-default.html、template-male.html、template-universal.html）
  - 重建 `initDefaultTemplate.js` 为多模板播种器（幂等，启动时自动比对并更新数据库）
  - 在 `cardGenerator.js` 中添加 `{{position}}` 和 `{{birthday}}` 占位符支持，生日格式化为 "M月D日"
  - 在 `app.js` 中恢复种子调用，每次启动自动同步模板文件到数据库

## 三、安全评估

### 已修复的安全问题

| 问题 | 修复方式 |
|------|----------|
| Mass Assignment 漏洞 | 白名单过滤 sanitizeInput / sanitizeEmployeeInput |
| 路径遍历攻击 | UUID 正则验证 + path.basename + path.resolve |
| JWT_SECRET 默认值 | 生产环境检测并警告 |
| records 路由缺少认证 | 添加 authMiddleware |
| 贺卡ID可预测 | 使用 UUID v4 |
