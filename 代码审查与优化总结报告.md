# 生日彩信系统 - 代码审查与优化总结报告

## 一、当前项目技术架构概述

### 1.1 整体架构
```
生日彩信/
├── server/                     # 后端 (Node.js + Express + ES Modules)
│   ├── src/
│   │   ├── app.js              # 应用入口 + 启动逻辑
│   │   ├── config/             # 配置 (数据库、应用)
│   │   ├── middlewares/        # 中间件 (认证、错误处理)
│   │   ├── models/             # 数据模型 (Sequelize ORM)
│   │   ├── routes/             # API路由 (auth, employees, templates, records, card)
│   │   ├── services/           # 业务服务 (调度器, 贺卡生成, 模板匹配, 短信)
│   │   ├── utils/              # 工具函数 (JWT, 密码, 响应格式, Excel解析, 初始化)
│   │   └── data/               # 静态数据 (默认模板HTML)
│   ├── generated-cards/        # 生成的贺卡HTML文件
│   ├── uploads/                # Excel上传临时目录
│   └── .env                    # 环境变量配置
├── admin-web/                  # 前端 (Vue 3 + Element Plus, 待开发)
└── birthday_h5_rebuild.html    # 贺卡模板参考文件
```

### 1.2 技术栈
| 层次 | 技术 |
|------|------|
| 运行时 | Node.js (ES Modules) |
| Web框架 | Express 5.x |
| ORM | Sequelize 6.x |
| 数据库 | MySQL (mysql2) |
| 认证 | JWT (jsonwebtoken + bcryptjs) |
| 定时任务 | node-cron |
| 文件处理 | Multer (上传), xlsx (Excel解析), uuid (ID生成) |

### 1.3 数据库模型关系
```
Admin (管理员)
  └── hasMany → SendRecord (发送记录)

Employee (员工)
  ├── belongsTo → Template (默认模板, alias: default_template)
  └── hasMany → SendRecord

Template (模板)
  ├── hasMany → Employee
  └── hasMany → SendRecord

SendRecord (发送记录)
  ├── belongsTo → Admin (alias: admin)
  ├── belongsTo → Employee (alias: employee)
  └── belongsTo → Template (alias: template)
```

```
公开接口:

认证接口:

  GET  /api/auth/profile            # 获取当前用户信息
  POST /api/auth/change-password     # 修改密码

需认证接口:
  # 员工管理

  PUT    /api/templates/:id          # 修改模板
  DELETE /api/templates/:id          # 删除模板
  GET    /api/templates/:id/preview  # 模板预览

  # 发送记录

  POST   /api/records/test-send/:id  # 测试发送
```

## 二、已识别的问题及修复方案

### 2.1 关键问题 (已修复)

#### 问题1: SendRecord模型缺少 `admin_id` 字段
- **严重程度**: 高 (数据丢失)
- **描述**: `models/index.js` 定义了 `SendRecord.belongsTo(Admin, { foreignKey: 'admin_id' })`，且 `employees.js` 手动生成贺卡时写入 `admin_id`，但 `SendRecord` 模型定义中没有 `admin_id` 字段，Sequelize 会静默丢弃该数据
- **修复**: 在 `SendRecord.js` 模型中添加了 `admin_id` 字段定义

#### 问题2: records路由缺少认证中间件
- **严重程度**: 高 (安全漏洞)
- **描述**: 其余路由（employees, templates）都使用了 `router.use(authMiddleware)`，唯独 `records.js` 遗漏，导致发送记录和统计数据无需认证即可访问
- **修复**: 在 `records.js` 路由中添加 `router.use(authMiddleware)`

#### 问题3: response.js 的 `error()` 函数参数设计混乱
- **严重程度**: 高 (数据错误)
- **描述**: 旧签名 `error(res, message, code, statusCode)` 四个参数容易混淆，`records.js` 中调用 `error(res, '员工不存在', 1, 404)` 导致业务码为1而非404，`employees.js` 中 `error(res, '部分验证失败', 400, {...})` 将对象传入了statusCode位置
- **修复**: 简化为 `error(res, message, statusCode, extraData)` 三个核心参数，code统一使用statusCode

#### 问题4: 调度器创建失败记录时 `template_id: null` 
- **严重程度**: 高 (数据库约束违反)
- **描述**: `scheduler.js` 异常处理中创建失败记录时 `template_id: null`，但模型定义 `allowNull: false`，会导致数据库约束错误
- **修复**: 重构异常处理逻辑，失败时仅记录日志，不创建不完整的数据库记录

#### 问题5: 模板CRUD存在Mass Assignment漏洞
- **严重程度**: 高 (安全漏洞)
- **描述**: `Template.create(req.body)` 和 `Template.update(req.body)` 直接将请求体传递给数据库操作，攻击者可通过请求篡改任意字段（id, timestamps等）
- **修复**: 添加 `sanitizeInput()` 白名单过滤函数，仅允许 `TEMPLATE_FIELDS` 中定义的字段通过

#### 问题6: 贺卡路由存在路径遍历风险
- **严重程度**: 高 (安全漏洞)
- **描述**: card路由仅对cardId做了长度检查(>=10)，未防范 `../` 等路径遍历字符
- **修复**: 使用UUID正则验证格式 + `path.basename()` 提取纯文件名 + `path.resolve()` 比较防止目录穿越

### 2.2 中等优先级问题 (已修复)

#### 问题7: 多处重复 dotenv.config() 调用
- **描述**: `database.js`, `config/index.js`, `app.js` 三处都调用了 `dotenv.config()`
- **修复**: 采用"条件式惰性加载"模式，`app.js` 在最顶部统一调用，其他模块仅在环境变量未加载时回退调用

#### 问题8: employees.js使用动态import获取SendRecord
- **描述**: `const { SendRecord } = await import('../models/index.js')` 是多余的动态导入，其他路由都是顶部静态导入
- **修复**: 改为顶部静态导入 `import { ..., SendRecord } from '../models/index.js'`

#### 问题9: Excel导入使用同步文件操作(fs.unlinkSync)
- **描述**: `fs.unlinkSync()` 同步删除文件可能阻塞事件循环
- **修复**: 改为异步 `fs/promises.unlink()`，并增加 try/catch 避免删除失败时崩溃

### 2.3 功能增强 (已实施)

#### 增强1: 调度器防重复发送
- **描述**: 原调度器每天8点对所有生日员工执行，若因故障重启会重复发送
- **修复**: 在发送前查询当天已成功发送的员工ID集合，自动跳过已发送的

#### 增强2: scheduler核心逻辑函数化
- **描述**: `processBirthdayEmployees()` 提取为独立导出函数，便于手动触发和单元测试

#### 增强3: 员工CRUD添加输入白名单
- **描述**: 员工新增/修改也添加了 `sanitizeEmployeeInput()` 白名单过滤，防止字段篡改

#### 增强4: cardGenerator缺少目录创建
- **描述**: 如果 `generated-cards` 目录不存在，`writeFile` 会失败
- **修复**: 添加目录不存在时自动创建的逻辑

## 三、配置项和参数可调整说明

### 3.1 环境变量 (.env)

| 变量名 | 默认值 | 说明 | 建议 |
|--------|--------|------|------|
| `DB_HOST` | localhost | MySQL地址 | 生产环境需修改 |
| `DB_PORT` | 3306 | MySQL端口 | - |
| `DB_NAME` | birthday | 数据库名 | - |
| `DB_USER` | root | 数据库用户 | 生产环境使用专用账户 |
| `DB_PASSWORD` | (必填) | 数据库密码 | **不要使用root/空密码** |
| `JWT_SECRET` | (默认) | JWT签名密钥 | **必须修改为随机长字符串** |
| `JWT_EXPIRES_IN` | 24h | Token过期时间 | 可按需调整 |
| `PORT` | 3000 | 服务器端口 | - |
| `NODE_ENV` | development | 运行环境 | 生产环境设为production |
| `CARDS_DIR` | ./generated-cards | 贺卡存储目录 | 建议使用绝对路径 |
| `UPLOADS_DIR` | ./uploads | 上传临时目录 | 建议使用绝对路径 |
| `BASE_URL` | http://localhost:3000 | 贺卡访问基础URL | **生产环境必须修改** |

### 3.2 数据库连接池 (config/database.js)

```javascript
pool: {
  max: 10,        // 最大连接数，生产可调至20-50
  min: 0,         // 最小连接数
  acquire: 30000, // 获取连接超时(ms)
  idle: 10000     // 空闲连接回收时间(ms)
}
```

### 3.3 定时任务 (services/scheduler.js)

```javascript
cron.schedule('0 8 * * *', async () => {  // 每天8:00
```
- 可通过修改cron表达式调整执行时间
- 时区固定在 `Asia/Shanghai`

### 3.4 模板匹配逻辑 (services/templateMatcher.js)

匹配优先级：**手动指定 > 年龄+性别 > 性别 > 通用模板**
- 年龄计算基于生日精确到天
- 可通过 `templates` 表的 `match_age_min/max` 设定年龄段

## 七、维护注意事项

### 7.1 数据库变更
- 新增字段后，确保不在生产环境使用 `sequelize.sync({ alter: true })`
- 推荐使用 Sequelize Migration 管理数据库版本

### 7.2 文件系统
- `generated-cards/` 和 `uploads/` 需要定期清理
- `uploads/` 中可能有Excel导入失败的残留文件
- 建议添加磁盘空间监控

### 7.3 定时任务
- 调度器失败不会导致服务器崩溃
- 建议添加健康检查和失败告警（钉钉/企业微信通知）

### 7.4 安全建议
- 生产环境务必修改 JWT_SECRET 和默认管理员密码
- 建议启用 HTTPS
- 贺卡ID使用UUID v4确保不可预测性
